# C++クラスメソッド修正時の影響範囲調査ガイド

---

## 1. 本資料の目的・定義・技術的前提・理論的根拠

### 1.1 目的

C++言語において、クラスのメソッドを修正した場合に、その変更が「最終的にビルドされるバイナリ」に影響を与える範囲を調査し、「再テストしなければならない範囲」を**漏れなく特定する**ための調査手順を示す。

### 1.2 用語の定義

本資料では以下の用語を定義する。

- **影響範囲**とは、変更された定義を利用する箇所の集合である
- 意味的に同じかどうか、実行結果が変わるかどうかは影響範囲の定義に含めない
- **再テスト範囲**とは、影響範囲を出発点として、実行時の振る舞いへの影響可能性を考慮して導出したテスト対象の集合である
- 影響範囲 ⊇ 再テスト範囲 の関係が常に成り立つ

### 1.3 技術的前提

本資料は以下の技術的前提に基づく。

- **clangd**は各ソースファイルを、そのファイルに対応するcompile commandで独立にインデックスする。本資料における「参照の検索」はclangdのインデックス結果に基づく
- **compile_commands.json**が完全である（全ソースファイルの全コンパイル条件のエントリが存在することを前提とする）
- compile_commands.jsonのコンパイル条件から外れた部分のソースコードはコンパイルされないため、その外れた部分のソースコードの修正はビルド後のバイナリに影響はない

### 1.4 二段階モデルの概要

本資料は、プログラムスライシング理論（Weiser, 1984）、変更影響分析（Bohner & Arnold, 1996）、および回帰テスト選択技法（Rothermel & Harrold, 1996）に基づき、調査を以下の二段階で行う。

**第一段階：影響範囲の特定（構造的・網羅的）**

- 1.2節で定義した影響範囲を特定する
- この段階では安全側に過大近似を取り、見落としを防ぐ

**第二段階：再テスト範囲の導出（実務的・絞り込み）**

- 1.2節で定義した再テスト範囲を導出する
- この段階で初めて、意味的判断（判定結果が変わるか、本番動作に影響するか等）を用いて絞り込みを行う

この二段階の分離により、構造的な網羅性を保証しつつ、過剰テストを避ける実務的な絞り込みを可能にする。第2.3節で述べる「再テスト範囲からの除外条件」は第二段階の絞り込みに該当し、影響範囲そのものの縮小ではない。

### 1.5 三つの調査区分（A・B・C）

影響範囲の調査を以下の三区分に分類する。

**区分A：コード依存の影響分析**

前方スライス（Forward Slice）に基づき、データ依存と制御依存をたどってコード上の影響範囲を追跡する。これはプログラミング言語を問わず適用可能な、影響分析の基本骨格である。波及方向（上・横・下）の追跡、関数境界を越える漏洩チェック、および再テスト範囲の絞り込み（影響範囲から再テスト対象外を除外する判定）を含む。

**区分B：C++言語固有の影響分析**

C++の言語特性に起因する、前方スライスだけでは捕捉しきれない影響経路を追跡する。仮想関数と動的ディスパッチ、テンプレート/inline/header/ODRの波及、エイリアス/所有権/寿命、例外/noexcept/契約、およびビルド/リンク/ABIを含む。

**区分C：コード外の影響分析**

コードの依存関係だけでは検出できない影響を追跡する。外部I/O/永続化、並行性/タイミング、非機能要件（性能・メモリ等）、ビルド設定/条件コンパイル差分、およびテスト資産そのものの調査を含む。

**区分間の関係**：区分Aが分析の基本骨格であり、区分Bがその上にC++固有の拡張を加え、区分Cがコード外の観点で補完する。漏れのない調査のためには三区分すべてを実施する必要があるが、変更の性質によって各区分の調査深度は異なる（第6章で詳述）。

### 1.6 理論的根拠：プログラムスライシング

プログラム中のある文を起点として、**データ依存**と**制御依存**をすべてたどって到達可能な文の集合を「スライス」と呼ぶ。

- **後方スライス（Backward Slice）**：ある変数の値がどの文の影響で決まったかを遡る
- **前方スライス（Forward Slice）**：ある文を変更した場合にどの文が影響を受けるかを追跡する

**影響範囲の調査における主要な分析骨格として、前方スライスを用いる。**

前方スライスに含まれない文は、修正箇所とのデータ依存も制御依存も存在しないため、修正の影響を受ける可能性が極めて低い。前方スライスは、影響範囲を体系的に追跡するための最も確立された理論的枠組みである。

### 1.7 理論的限界：静的解析は近似である

ただし、以下の限界を認識しておく必要がある。

**（1）一般のプログラムにおける完全判定は理論上不可能**

プログラムの意味的性質は一般にundecidable（決定不能）であり、あらゆる静的解析は近似にならざるを得ない。信頼性を重視するsound（健全）な解析は、偽陰性（見落とし）を避ける代わりに偽陽性（過剰検出）を含む過大近似を返す。

**（2）C++特有の困難**

C++では、ポインタエイリアス、動的ディスパッチ、テンプレート実体化、例外伝播などにより、静的な前方スライスだけでは影響の全容を捕捉しきれない場合がある。これが区分Bを必要とする理由である。

**（3）非機能的影響はスライスの追跡対象外**

同じコードパスを通る場合でも、アルゴリズム変更による性能劣化、ロック順序変更によるデッドロック、タイミング変化によるレースコンディションの顕在化など、データフロー/制御フローの依存関係だけでは検出できない影響が存在する。これが区分Cを必要とする理由である。

**したがって本資料では、前方スライスを「主要な分析骨格」（区分A）として使用しつつ、スライスだけでは捕捉できない影響経路を区分B・Cとして補完する。漏れゼロを目指す場合は、安全側の過大近似で候補を出し、不確実な変更では再テスト範囲を拡大するのが原則である。**

### 1.8 前方スライスが追跡する依存関係の種類

前方スライスが追跡する依存関係は以下の2種類であり、コード上の影響伝播の基本経路をカバーする。

**（1）データ依存（Data Dependence）**

ある文の出力（変数への代入）が、別の文の入力として使用される関係。

```cpp
int a = foo.calculate();   // S1：起点
int b = a * 2;             // S2：S1にデータ依存（aを使用）
network.send(b);           // S3：S2にデータ依存（bを使用）
```

**（2）制御依存（Control Dependence）**

ある文の結果によって、別の文が実行されるかどうかが決まる関係。

```cpp
if (b > threshold) {       // S4：S2にデータ依存
    doA();                 // S5：S4に制御依存（S4の結果で実行が決まる）
} else {
    doB();                 // S6：S4に制御依存（S4の結果で実行が決まる）
}
```

これら2種類の依存関係はコード上の影響伝播の基本であるが、C++の実務ではこれに加えて、エイリアス経由の間接依存、動的ディスパッチ経由の依存（区分B）、外部I/O経由の依存、非機能特性への影響（区分C）なども追跡対象にする必要がある。

### 1.9 前方スライスの具体例

```cpp
void example() {
    Foo f;
    int a = f.calculate();   // [S1] 修正箇所（起点）
    int b = a * 2;           // [S2] S1にデータ依存
    int c = 100;             // [S3] 起点と依存関係なし
    network.send(b);         // [S4] S2にデータ依存
    logger.log(c);           // [S5] S3にデータ依存（起点とは無関係）
    if (b > 10) {            // [S6] S2にデータ依存
        doA();               // [S7] S6に制御依存
    } else {
        doB();               // [S8] S6に制御依存
    }
}
```

前方スライスの結果：

| 文 | スライスに含まれるか | 理由 | 再テスト |
|---|---|---|---|
| S1 | ─ | 修正起点 | 必要 |
| S2 | 含まれる | S1のデータ依存 | **必要** |
| S3 | **含まれない** | S1との依存なし | 不要（注） |
| S4 | 含まれる | S2のデータ依存 | **必要** |
| S5 | **含まれない** | S3のみに依存 | 不要（注） |
| S6 | 含まれる | S2のデータ依存 | **必要** |
| S7 | 含まれる | S6の制御依存 | **必要** |
| S8 | 含まれる | S6の制御依存 | **必要** |

（注）「不要」はコード依存（区分A）の観点での判定である。非機能特性（性能劣化等）が懸念される変更の場合は、スライス外の文であっても影響確認が必要な場合がある（区分C：第4章参照）。

## 2. 区分A：コード依存の影響分析

本章では、前方スライスに基づくコード上の影響追跡を扱う。これは影響分析の基本骨格であり、プログラミング言語を問わず適用可能な手法である。

### 2.1 影響の波及方向

前方スライスの追跡は、関数の内部に閉じるものではない。影響は**3つの方向**に波及しうる。

#### 2.1.1 上方向：呼び出し元への伝播

修正メソッドの呼び出し元に対して、戻り値や副作用が伝播するケース。

```cpp
int caller() {
    Foo f;
    int result = f.calculate();  // 修正メソッドの呼び出し
    return result;               // 戻り値として上位に伝播
}

void upper() {
    int r = caller();       // ← 影響を受ける（再テスト対象）
    report.submit(r);       // ← 影響を受ける（再テスト対象）
}
```

#### 2.1.2 横方向：戻り値の別関数への引数渡し

修正メソッドの戻り値が、別の関数やメソッドの引数として渡されるケース。

```cpp
void caller() {
    Foo f;
    int result = f.calculate();
    a.methodA(result);           // ← 横方向への波及（再テスト対象）
    b.methodB(result);           // ← 横方向への波及（再テスト対象）
    c.methodC(result);           // ← 横方向への波及（再テスト対象）
}
```

横方向の波及パターンは以下を含む：

**パターン1：直接引数として渡す**

```cpp
network.send(foo.calculate());   // sendの動作が変わりうる → 再テスト
```

**パターン2：加工してから渡す**

```cpp
int val = foo.calculate();
int adjusted = val * 2 + offset;
db.store(adjusted);              // データ依存が続く限り影響は伝播 → 再テスト
```

**パターン3：条件分岐に使われる**

```cpp
int val = foo.calculate();
if (val > threshold) {
    doA();     // 分岐結果が変わりうる → 再テスト
} else {
    doB();     // 分岐結果が変わりうる → 再テスト
}
```

**パターン4：複数箇所に渡される（fan-out）**

```cpp
int val = foo.calculate();
a.methodA(val);   // ← 再テスト
b.methodB(val);   // ← 再テスト
c.methodC(val);   // ← 再テスト
```

#### 2.1.3 下方向：呼び出し先の内部への伝播

修正メソッドの戻り値を受け取った関数の内部処理にも影響は伝播する。

```cpp
void A::methodA(int val) {
    int x = val + 10;     // データ依存で影響を受ける
    db.store(x);          // さらに伝播する → 再テスト対象
}
```

#### 2.1.4 波及方向の全体像

```
                  呼び出し元（upper）
                      ▲
                      │ 上方向（戻り値の伝播）
                      │
    methodA() ◀── calculate() ──▶ methodB()
                      │
                      │ 横方向（引数としての伝播）
                      │
                  methodC()
                      │
                      ▼ 下方向（呼び出し先の内部への伝播）
                  methodCの内部処理
```

前方スライスはこれらすべての方向を区別せず、依存関係をたどって網羅的に追跡する。

### 2.2 関数境界を越える影響の追跡（関数外への漏洩チェック）

関数の内部だけで影響が閉じているかどうかを判定する必要がある。修正の影響が関数の外に「漏れ出す」経路は以下の5つである。**すべてを確認しなければならない。**

#### 2.2.1 経路①：戻り値

```cpp
int example() {
    Foo f;
    int a = f.calculate();
    int b = a * 2;
    return b;  // ← 修正の影響が戻り値として外に出る
}

void upper() {
    int r = example();      // ← 影響を受ける
    report.submit(r);       // ← 影響を受ける
}
```

**確認事項**：戻り値に修正メソッドの影響が含まれているか。void関数であっても、参照で値を返していないかを確認する。

#### 2.2.2 経路②：メンバ変数への書き込み

```cpp
class MyClass {
    int cachedValue;
public:
    void example() {
        Foo f;
        cachedValue = f.calculate();  // ← メンバ変数に書き込み
    }

    void other() {
        doSomething(cachedValue);     // ← 同じメンバを読む → 影響を受ける
    }
};
```

**確認事項**：修正の影響を受けた値がメンバ変数に格納され、他のメソッドから読み取られていないか。

#### 2.2.3 経路③：参照・ポインタ引数への書き込み

```cpp
void example(int& outVal) {
    Foo f;
    outVal = f.calculate();  // ← 参照引数に書き込み
}

void upper() {
    int val;
    example(val);           // valが変わる
    printer.print(val);     // ← 影響を受ける
}
```

**確認事項**：参照渡しまたはポインタ渡しの引数に、修正の影響を受けた値が書き込まれていないか。

#### 2.2.4 経路④：グローバル変数・静的変数への書き込み

```cpp
static int globalCounter;

void example() {
    Foo f;
    globalCounter = f.calculate();  // ← グローバルに書き込み
}

// 全く別の場所（任意のファイル、任意の関数）
void unrelated() {
    if (globalCounter > 0) {  // ← 影響を受ける
        doSomething();
    }
}
```

**確認事項**：修正の影響を受けた値が、グローバル変数・静的変数・シングルトンなどに書き込まれていないか。この経路が存在する場合、影響範囲はプログラム全体に広がりうるため、最も注意を要する。

#### 2.2.5 経路⑤：コールバック・observer・signal-slot経由の副作用

```cpp
void example() {
    Foo f;
    int a = f.calculate();
    eventBus.emit("valueChanged", a);  // ← 登録済みリスナー全員に影響
}
```

**確認事項**：修正の影響を受けた値が、コールバック呼び出し、observer通知、signal-slot機構、イベントバスなどの間接的な通知経路を通じて伝播していないか。

#### 2.2.6 間接的な漏洩（呼び出し先経由の漏洩）

関数内で直接的な漏洩がなくても、呼び出し先の内部で漏洩が発生している場合がある。

```cpp
void example() {
    Foo f;
    int a = f.calculate();
    network.send(a);       // ← 直接的にはexampleから外に出ていないように見える
}

// しかしsendの内部で...
void Network::send(int val) {
    lastSentValue = val;   // ← メンバ変数に書き込み（漏洩！）
}

void upper() {
    example();
    int v = network.getLastSent();  // ← 影響を受ける
}
```

**確認事項**：前方スライスに含まれるすべての呼び出し先関数について、経路①〜⑤で外部に影響が漏洩していないかを再帰的に確認する。

#### 2.2.7 関数外への漏洩判定フロー

```
修正メソッドの呼び出し元関数 example() について：

  ├── 戻り値に影響が含まれるか？            ──YES──▶ 呼び出し側も調査
  ├── メンバ変数に影響が書き込まれるか？      ──YES──▶ 呼び出し側も調査
  ├── 参照/ポインタ引数に書き込まれるか？     ──YES──▶ 呼び出し側も調査
  ├── グローバル/静的変数に書き込まれるか？    ──YES──▶ プログラム全体を調査
  ├── コールバック/observer等に通知されるか？  ──YES──▶ 全リスナーを調査
  ├── 呼び出し先の内部で上記が起きるか？      ──YES──▶ 該当経路も調査
  └── すべてNO                              ──────▶ この関数の外には影響が及ばない
```

### 2.3 再テスト範囲からの除外条件

影響範囲（変更された定義を利用する箇所の集合）に含まれる箇所であっても、以下の条件に該当する場合は、再テスト範囲から除外できる可能性がある。これは二段階モデル（1.4節）の第二段階に該当し、影響範囲そのものの縮小ではなく、再テスト範囲への絞り込みである。

**重要**：以下の除外判定は意味的判断を伴うため、判断を誤ると見落としにつながる。不確実な場合は安全側に再テスト範囲に含める。

| 除外条件 | 理由 | 適用時の注意 |
|---|---|---|
| 戻り値が使われていない（discardされている） | データ依存が後続に発生しない | 副作用がないことの確認が必要 |
| 結果がbool判定等に変換され、判定結果が変わらない | 実行時の振る舞いが変化しない | 値域の変化を慎重に検証する必要がある |
| ログ出力のみに使われている | 本番動作の実行パスに影響しない | ログ内容の正しさを問う場合は除外不可 |
| assert/デバッグ用途のみ | 本番ビルドに含まれない | Releaseビルドで除去されることの確認が必要 |
| 関数内部で影響が閉じており、外部への5経路（2.2節）がすべて該当しない | 関数境界で再テスト対象が限定される | 呼び出し先内部の間接漏洩（2.2.6節）を確認済であること |

**注意**：上記はコード上のデータ依存・制御依存に基づく除外条件である（区分A）。非機能特性（性能劣化、タイミング変化等）が懸念される変更の場合、上記条件に該当してもなお再テスト対象とすべき場合がある（区分C：第4章参照）。

**影響範囲としての記録**：上記条件により再テスト範囲から除外した箇所であっても、影響範囲には含まれる。調査報告書（第7章）には、影響範囲に含まれるが再テスト範囲からは除外した箇所と、その除外根拠を記録する。

## 3. 区分B：C++言語固有の影響分析

本章では、C++の言語特性に起因する、前方スライス（区分A）だけでは捕捉しきれない影響経路を扱う。

### 3.1 仮想関数・ポリモーフィズムと動的ディスパッチ

仮想関数をオーバーライドしている場合、基底クラスのポインタ・参照経由の呼び出しもすべて影響範囲に含まれる。OOの変更影響分析において、継承と動的ディスパッチは非局所影響の主要因とされている。

```cpp
class Base {
public:
    virtual int calculate();
};

class Derived : public Base {
public:
    int calculate() override;  // ← これを修正
};

void process(Base* p) {
    p->calculate();  // 実行時にDerivedの可能性がある → 再テスト対象
}
```

**調査手順**：

1. 修正メソッドがvirtual/overrideであるかを確認する
2. 仮想関数である場合、基底クラスのポインタ・参照を通じた呼び出し箇所をすべて特定する
3. 派生クラスで同じ仮想関数をoverrideしている他のクラスにも影響がないか確認する
4. finalの有無で上書き可能性がどう変わるかを確認する
5. デストラクタのvirtual性に影響がないかを確認する（基底ポインタ経由のdeleteで未定義動作の危険）
6. それらの呼び出し箇所を前方スライスの起点に含める

### 3.2 テンプレート

テンプレートで使われるクラスのメソッドを修正した場合、テンプレートがインスタンス化されたすべての箇所が影響範囲に含まれる。

```cpp
template<typename T>
void process(T& obj) {
    auto result = obj.calculate();
    doSomething(result);
}

// 以下のすべてが影響範囲
process<Foo>(foo);      // ← 再テスト対象
process<Bar>(bar);      // ← Barがcalculate()を持つなら要確認
```

**追加確認事項**：特殊化・部分特殊化・SFINAE/conceptsへの影響、およびどのtranslation unitでimplicit instantiation/odr-useされるかも確認する（3.5節で詳述）。

### 3.3 演算子オーバーロード・コピー/ムーブセマンティクス

修正したメソッドが演算子オーバーロード経由で暗黙的に呼び出されていないかを確認する。暗黙の型変換演算子を通じた呼び出しにも注意が必要である。

修正メソッドの影響を受けたオブジェクトがコピーまたはムーブされた場合、コピー先・ムーブ先にも影響が波及する。コンストラクタ・デストラクタ・特殊メンバ関数との相互作用にも注意が必要である。

### 3.4 エイリアス・所有権・寿命

C++ではaliasingが強く、「どの名前で読むか」ではなく「同じ実体を共有しているか」で追跡する必要がある。前方スライスの理論的枠組み（区分A）だけでは、この間接的な共有を見落としやすい。

#### 3.4.1 参照・ポインタ経由の別名参照

```cpp
void example(Foo& ref) {
    ref.calculate();  // refとoriginalは同じ実体
}

Foo original;
Foo& alias = original;
example(alias);        // ← originalの状態が変わっている
useOriginal(original); // ← 影響を受ける
```

#### 3.4.2 スマートポインタ経由の共有

```cpp
auto shared = std::make_shared<Foo>();
auto copy = shared;  // 同一オブジェクトを共有
shared->calculate(); // ← 修正メソッド
copy->getValue();    // ← 同じ実体なので影響を受ける
```

#### 3.4.3 所有権とconst性

確認事項：
- 引数が参照かポインタか値渡しか
- const性を変えていないか
- 返り値が参照/ポインタ/ビューか
- 所有権を移す実装に変わっていないか（unique_ptr等）
- move/copyの発生条件が変わっていないか
- 一時オブジェクトやdangling参照の危険が増えていないか
- スマートポインタの共有・循環・解放タイミングが変わっていないか

#### 3.4.4 再テスト対象

- コピー/ムーブを伴うテスト
- リソース解放テスト
- 破棄順序や寿命に敏感なテスト
- ASan/UBSan/LSanなどの動的検査

### 3.5 template / inline / header / ODRの波及

C++で特に見落としやすい領域であり、前方スライスの理論（区分A）だけではカバーできない。

#### 3.5.1 ヘッダ定義とimplicit inline

C++では、クラス定義内のメソッドは通常implicit inlineであり、ヘッダファイルに定義が置かれる。このため、メソッド修正がそのヘッダをincludeするすべてのtranslation unit（TU）に波及する。

**確認事項**：
- メソッド定義がヘッダにあるか
- クラス定義内で定義されているか（implicit inline）
- inline / constexpr / consteval / deleted / defaulted により暗黙inlineか
- class template / function template / member templateに関係するか
- 特殊化・部分特殊化・SFINAE / conceptsに影響するか
- どのtranslation unitでodr-use / instantiationされるか
- include依存先がどこまで広がるか

#### 3.5.2 ODR（One Definition Rule）

ODR上、inline関数やtemplated entityは複数TUにまたがる前提があり、定義の変更はそれらすべてのTUに影響する。ODR違反を起こす変更は未定義動作になりうる。

#### 3.5.3 再テスト対象

- そのヘッダを含むモジュール群のビルド確認
- template実体化パターンごとのテスト
- 異なる型引数でのテスト
- 依存TUを含む回帰テスト

### 3.6 例外・noexcept・契約

#### 3.6.1 例外仕様の変更

修正メソッドの例外に関して、以下を確認する。

- 以前は成功していた入力で例外が出るようになっていないか
- 以前は例外だった入力で成功するようになっていないか
- 例外型が変わっていないか
- 例外境界でリソースリークしないか
- rollback / retry / fallbackの動きが変わっていないか
- noexcept違反が起きないか

noexceptは関数型に関係し、コンパイラ最適化やnoexcept演算子の結果にも影響するため、この変更は無視できない。修正メソッドがスローする例外が変わりうる場合、catch節の動作も影響を受ける。例外経路も前方スライスの追跡対象に含める。

#### 3.6.2 メソッド契約の確認

変更前後で以下の契約が維持されているかを確認する。

- 事前条件
- 事後条件
- クラス不変条件
- 戻り値の値域・意味
- エラーコードの意味
- 実行順序依存の副作用

#### 3.6.3 再テスト対象

- 例外系テスト
- rollback/recoveryテスト
- 異常系の統合テスト
- 失敗系・境界値テスト

### 3.7 ビルド・リンク・ABI

コード依存（区分A）とは別に、ビルドシステムとリンク結果への影響を確認する必要がある。この観点は前方スライスの理論には含まれないが、C++の実務では不可欠である。

**確認事項**：
- シグネチャ変更はあるか
- オーバーロード解決結果が変わるか
- publicクラスのレイアウトが変わるか（メンバ追加/削除/並べ替え）
- vtable / 仮想関数集合に変化があるか
- 共有ライブラリ・DLL・プラグイン境界を跨ぐか
- ODR影響がないか
- 別リポジトリや別成果物がこのAPI/ABIに依存していないか

**再テスト対象**：
- 依存ライブラリとの結合テスト
- ABI互換性確認
- プラグイン読み込みテスト
- 本番相当リンク構成でのスモークテスト

## 4. 区分C：コード外の影響分析

本章では、コードの依存関係（区分A・B）だけでは検出できない影響を扱う。前方スライス（データ依存＋制御依存）の追跡対象外であるが、「漏れなく」を目指すなら不可欠な調査である。

### 4.1 外部I/O・永続化

修正の影響がプログラム外部に出る場合、その外部境界を検証するテストまで再テスト対象に含める。

**確認事項**：
- DB書き込み内容やスキーマへの影響
- ファイル出力・シリアライズ形式への影響
- ネットワーク送信内容・順序・タイミングへの影響
- メッセージキューやイベントのpayloadへの影響
- 監査ログ・メトリクス・アラート条件への影響
- 再起動後や別プロセスで影響が見えるか

**再テスト対象**：
- DB/ファイル/通信の統合テスト
- シリアライズ互換性テスト
- 監査ログ確認テスト
- E2Eテスト

### 4.2 並行性・タイミング

スライスや静的依存だけでは、並行性やタイミングの不具合は拾い切れない。

**確認事項**：
- ロック取得順序を変えていないか
- 原子変数アクセス順序を変えていないか
- wait/notify条件を変えていないか
- タイムアウト・リトライ間隔を変えていないか
- 処理が速くなった/遅くなったことでraceが顕在化しないか
- スレッド間で観測される状態遷移が変わらないか

**再テスト対象**：
- 並行実行テスト
- race/deadlock検査（TSanなど）
- 負荷試験
- soak/stressテスト

### 4.3 非機能要件

同じコードパスを通る場合でも、非機能特性の劣化は発生しうる。

**確認事項**：
- レイテンシ
- スループット
- CPU使用率
- メモリ使用量・アロケーション回数
- 起動時間
- リアルタイム制約

**再テスト対象**：
- 性能回帰テスト
- メモリ回帰テスト
- 負荷・容量テスト

### 4.4 ビルド設定・条件コンパイル差分（変更の種類に応じて）

template/inline/header変更、最適化レベルに依存する変更、プラットフォーム固有の変更では、以下も確認する。

**前提**：本資料ではcompile_commands.jsonが完全である（全ソースファイルの全コンパイル条件のエントリが存在する）ことを前提とする（1.3節）。compile_commands.jsonのコンパイル条件から外れた部分のソースコードはビルド後のバイナリに含まれないため、調査対象外とする。以下の確認事項は、compile_commands.jsonに記録されたコンパイル条件の範囲内で実施する。

**確認事項**：
- compile_commands.jsonに含まれる各コンパイル条件で、変更箇所がコンパイルされるかを確認する
- Debug / Release差分（compile_commands.jsonに両方のエントリがある場合）
- OS / コンパイラ差分（同上）
- `#ifdef` / feature flag（各コンパイル条件下で有効/無効となるマクロ分岐の確認）
- 32/64bit
- C++標準バージョン差分

**再テスト対象**：
- 主要ビルドマトリクスでのスモーク
- 条件コンパイル有効/無効ケース
- 複数コンパイラでのビルドと最小実行確認

### 4.5 テスト資産そのものの調査

影響分析の対象はコードだけでなく、テスト資産の整合性まで含む。

**確認事項**：
- 既存の単体テストでこの変更をカバーしているか
- 既存の統合/E2Eテストで観測点をカバーしているか
- 期待値更新が必要なテストはどれか
- 新規テストが必要か
- obsoleteになったテストはないか

**参考（本資料のスコープ外）**：仕様書・設計書・レビュー観点の更新が必要かの確認は、バイナリへの影響範囲・再テスト範囲の特定とは別の活動であるが、変更管理プロセスとして併せて実施することが望ましい。

- [ ] ネットワーク送信内容・順序・タイミングへの影響を確認したか
- [ ] メッセージキュー・イベントpayloadへの影響を確認したか
- [ ] 監査ログ・メトリクス・アラート条件への影響を確認したか
- [ ] 再起動後や別プロセスで影響が見えるかを確認したか

#### STEP 10：並行性・タイミングの影響確認〔区分C〕

- [ ] ロック取得順序を変えていないか（4.2節）
- [ ] 原子変数アクセス順序を変えていないか
- [ ] wait/notify条件を変えていないか
- [ ] タイムアウト・リトライ間隔を変えていないか
- [ ] 処理速度変化によるrace顕在化の危険がないか
- [ ] スレッド間で観測される状態遷移が変わらないか

#### STEP 11：非機能要件の影響確認〔区分C〕

- [ ] レイテンシ・スループットへの影響を確認したか（4.3節）
- [ ] CPU使用率・メモリ使用量への影響を確認したか
- [ ] 起動時間・リアルタイム制約への影響を確認したか

### STEP 12：ビルド設定・条件コンパイル差分の確認〔区分C〕

- [ ] compile_commands.jsonに含まれる各コンパイル条件で、変更箇所がコンパイルされるかを確認したか（4.4節）
- [ ] template/inline/header変更に該当する場合、compile_commands.json内のビルド構成差分を確認したか（4.4節）
- [ ] Debug/Release差分、OS/コンパイラ差分を確認したか（compile_commands.jsonにエントリがある構成のみ）
- [ ] `#ifdef`/feature flag/32-64bitの影響を確認したか（各コンパイル条件下で有効/無効となる分岐のみ）

#### STEP 13：テスト資産の補足調査〔区分C〕

STEP 8で実施したテスト資産の照合を前提とし、区分Cの観点で追加の確認を行う。

- [ ] STEP 8で除外した外部観測点について、区分Cの観点（非機能的影響、タイミング変化等）でも除外が妥当かを確認する
- [ ] 区分A・Bでは検出されないが、区分Cの観点で追加が必要なテストがないか確認する
- [ ] obsoleteになったテストがないか確認する
- [ ] （参考・スコープ外）仕様書・設計書・レビュー観点の更新が必要か確認する

#### STEP 14：再テスト範囲の最終確認〔総合〕

A・B群（STEP 0〜8）およびC群（STEP 9〜13）の結果を統合し、再テスト範囲を最終確定する。

- [ ] STEP 8で確定した項目7（再テスト対象）に対し、C群の調査結果に基づく追加・変更がないか確認する
- [ ] 非機能的影響がSTEP 8の除外判定の範囲を超えて存在しないことを確認する
- [ ] 不確実性が残る箇所には、安全側に再テスト範囲を拡大する
- [ ] 項目5のすべての外部観測点が、項目7のいずれかの行に現れていることを最終確認する


## 6. 影響範囲の終点・変更レベル分類・結論

### 6.1 終点は「main」ではなく「外部観測点」

影響範囲の終点は、mainまで遡ることではなく、**外部から観測可能な仕様境界と、それを検証するテスト群**に到達することである。

外部観測点とは以下を指す：
- public APIの返り値・例外・事後状態
- 呼び出し側が読むメンバ状態
- グローバル/静的状態
- 永続化データ（DB、ファイル）
- ネットワーク/メッセージキュー出力
- ログ・メトリクス
- スレッド間共有状態
- UI/CLIの出力
- 性能・レイテンシ・メモリ量などの非機能値
- 上記の仕様を検証するテストケース

### 6.2 mainまで遡る必要がある場合・ない場合

| 状況 | mainまで遡る必要 |
|---|---|
| 戻り値が途中で使い捨てられている | **不要**（その時点でデータ依存が途切れる） |
| 副作用（メンバ変数変更等）がない修正 | **不要**（直接呼び出し元まで） |
| 途中の関数で影響が完全に閉じている | **不要**（その関数が境界） |
| 戻り値がmainまで連鎖的に伝播している | **必要** |
| グローバル変数やシングルトンを変更する副作用がある | **必要**（広範囲の調査が必要） |
| CLI/サーバ要求処理/バッチ処理全体の観測結果に届く | **必要**（E2Eテストまで対象） |
| その観測結果を検証する既存テストがE2E/システムテストしかない | **必要**（上位テストまで選択） |

結論として、追う単位は「main」ではなく「**影響が到達する外部観測点と、その観測点を検証するテスト**」である。

### 6.3 変更レベルに応じた調査範囲の目安

変更の性質に応じて、調査の深さを3段階に分類する。これは過剰テストを避けつつ、必要な範囲をカバーするための実務的な目安である。

#### レベルA：局所変更

**該当条件**：実装変更のみで、契約・例外・共有状態・virtual・template・外部I/O・性能のいずれにも影響なし。

**調査範囲**：
- 変更メソッドの単体テスト
- 直接呼び出し元の単体テスト
- 直接呼び出し元の統合テスト

**チェックリスト実施範囲**：A・B群のSTEP 0〜7を中心に実施。STEP 6は該当なしとして記録。C群は必要性を確認の上、省略可。

#### レベルB：中範囲変更

**該当条件**：戻り値の意味変更、共有状態変更、例外変更、外部I/O変更、性能影響ありのいずれかに該当。

**調査範囲**：
- 変更メソッドの単体テスト
- 直接/間接呼び出し元の単体テスト
- 影響が到達する外部観測点の統合テスト
- 関連シナリオの上位テスト

**チェックリスト実施範囲**：A・B群のSTEP 0〜7、およびC群のSTEP 8〜13をすべて実施。

#### レベルC：広域変更

**該当条件**：virtual/override、template/inline/header、ABI、並行性、永続化形式、共有ライブラリ境界のいずれかに関与。

**調査範囲**：
- 安全側に広げて再ビルド
- 広域回帰テスト
- 必要なE2Eテスト
- ビルドマトリクスでのスモークテスト

**チェックリスト実施範囲**：A・B群のSTEP 0〜7、およびC群のSTEP 8〜13をすべて実施し、不確実な箇所では安全側に範囲を拡大。

### 6.4 結論：漏れゼロを目指すための原則

1. **前方スライスで到達しないから不要**ではなく、**影響候補から漏れないように安全側に含める**のが正しい姿勢である
2. 打ち切り点はmainではなく、外部観測点とそれを検証するテストである
3. ソース依存（区分A・B）だけでなく、実行時依存・テスト資産依存（区分C）まで見る必要がある
4. 不確実性が残る変更では、安全側に再テスト範囲を拡大する

### 8.1 学術文献

**[1] Weiser, M. (1984). "Program Slicing." IEEE Transactions on Software Engineering, Vol. SE-10, No. 4, pp. 352-357.**

プログラムスライシングの原論文。データフローと制御フローの分析によりプログラムを自動分解する手法を提唱。前方スライス・後方スライスの定義および形式的な性質を示した。本資料における前方スライスに基づくコード依存追跡（区分A）の理論的基盤。

- IEEE Xplore: https://ieeexplore.ieee.org/document/5010248
- DOI: https://doi.org/10.1109/TSE.1984.5010248
- ACM Digital Library: https://dl.acm.org/doi/10.1109/TSE.1984.5010248

**[2] Bohner, S. A. & Arnold, R. S. (1996). "Software Change Impact Analysis." IEEE Computer Society Press.**

変更影響分析を体系化した書籍。SIS（Starting Impact Set）とEIS（Estimated Impact Set）の概念を定義し、呼び出しグラフとデータ依存グラフによるEIS算出手順を確立。影響分析の対象はコードだけでなく、開発文書・テスト文書まで含むことを明示。本資料の影響追跡手順の枠組みはこのEIS算出手順に基づく。

- Wiley: https://www.wiley.com/en-us/Software+Change+Impact+Analysis-p-9780818673849
- ISBN: 978-0-818-67384-9

**[3] Rothermel, G. & Harrold, M. J. (1996). "Analyzing Regression Test Selection Techniques." IEEE Transactions on Software Engineering, Vol. 22, No. 8, pp. 529-551.**

回帰テスト選択技法の評価枠組みを提示。変更前後の制御フローグラフを比較し、差分が通過するパスに関連するテストだけを選択すれば「safe」（変更により結果が変わり得る既存テストを漏らさない）であることを定義。本資料の再テスト範囲確定の考え方の根拠。ただし、safeの定義は「変更により結果が変わり得るテストを漏らさない」という意味であり、「スライス外の文は再テスト不要」という証明とは異なる点に注意。

- IEEE Xplore: https://ieeexplore.ieee.org/document/536955
- DOI: https://doi.org/10.1109/32.536955
- ACM Digital Library: https://dl.acm.org/doi/10.1145/248233.248262

**[4] Rothermel, G. & Harrold, M. J. (1997). "A Safe, Efficient Regression Test Selection Technique." ACM Transactions on Software Engineering and Methodology (TOSEM), Vol. 6, No. 2, pp. 173-210.**

[3]の手法を洗練し、安全かつ効率的な回帰テスト選択アルゴリズムを提示。

- ACM Digital Library: https://dl.acm.org/doi/10.1145/248233.248262

**[5] Gallagher, K. B. & Lyle, J. R. (1991). "Using Program Slicing in Software Maintenance." IEEE Transactions on Software Engineering, Vol. 17, No. 8, pp. 751-761.**

プログラムスライシングをソフトウェア保守に応用し、変更の影響を単一のスライス構成要素に分離する手法を提唱。

- ACM Digital Library: https://dl.acm.org/doi/10.1109/32.83912

**[6] Tip, F. (1995). "A Survey of Program Slicing Techniques." Journal of Programming Languages, Vol. 3, No. 3, pp. 121-189.**

プログラムスライシング技法の包括的なサーベイ。手続き間スライシング（interprocedural slicing）を含む各種手法を比較。

**[7] Horwitz, S., Reps, T., & Binkley, D. (1990). "Interprocedural Slicing Using Dependence Graphs." ACM Transactions on Programming Languages and Systems, Vol. 12, No. 1, pp. 26-60.**

手続き間（関数境界を越える）スライシングのアルゴリズムを提示。本資料の第2.2節「関数境界を越える影響の追跡」の理論的基盤。

**[8] Ryder, B. G. & Tip, F. (2001). "Change Impact Analysis for Object-Oriented Programs." ACM SIGPLAN-SIGSOFT Workshop on Program Analysis for Software Tools and Engineering (PASTE).**

OOプログラムにおける変更影響分析を扱い、継承・サブタイピング・動的ディスパッチが非局所影響の主要因であることを示した。既存クラスへのメソッド追加でさえvirtual callのディスパッチ挙動に影響し得ると指摘。本資料の第3.1節「仮想関数と動的ディスパッチ」の根拠。

### 8.3 C++言語規格

**[15] ISO/IEC 14882 (C++ Standard)**

本資料で言及するC++固有の概念（virtual, override, final, inline, constexpr, template, noexcept, ODR等）の正式な定義はこの規格による。特にODR（One Definition Rule）、implicit inline、noexceptの関数型への影響、テンプレートの実体化規則は、影響範囲調査（区分B）において重要な言語仕様である。
