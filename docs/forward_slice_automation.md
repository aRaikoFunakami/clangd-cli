# 「コード変更の影響範囲、完全に自動で追えますか？」—— 前方スライス自動化の技術的限界と現実的選択

*関数ひとつ変えたとき、その影響はどこまで及ぶのか。この問いに正確に答えることは、数学的に不可能であることが証明されている。本稿では、この「不可能性」がどこから来るのか、現実のツールはどう折り合いをつけているのか、そしてChromiumクラスの巨大C++コードベースでなぜ特に困難なのかを、エビデンスとともに解説する。*

## 本資料の位置づけ

[影響範囲調査ガイド](./impact_analysis_guide.md)では、前方スライス（Forward Slice）を影響分析の基本骨格として用いている。本資料は、前方スライスの自動化に関する技術的限界と、clangd-cliプロジェクトにおける現実的な選択について整理する。

## この記事の背景

ソフトウェア開発において「この関数を修正したら、どこに影響が及ぶか」を特定する作業——影響範囲調査——は、あらゆるコードレビューやリファクタリングの前提となる。

この調査の基本骨格となるのが**前方スライス（Forward Slice）** という技法だ。ある文（statement）を起点に、「その文の出力値がプログラム内のどの文に伝播するか」をデータの流れと制御の流れに沿って追跡するものである。

我々のclangd-cliプロジェクトでは、影響範囲調査を以下のように分業している。

- **データ収集（自動化済）**：clangd-cliの`investigate`コマンドが、clangdのLSP APIを通じて呼び出しグラフ・型階層・仮想関数ディスパッチ等の構造情報を一括取得する
- **前方スライス判定（LLMが担当）**：取得した各呼び出し元について、影響が実際に伝播するかどうかをLLMがコードを読んで判定する

本稿の問いは単純だ。**前方スライスの判定も自動化できないのか？** 答えは「理論的に完全な自動化は不可能であり、実用的な近似にも深刻な障壁がある」だ。以下、その理由をエビデンスとともに掘り下げていく。

## 自動化できている部分：「誰が誰を呼んでいるか」

まず、すでに自動化されている部分を明確にしておこう。clangd-cliの`investigate`コマンドは、clangdのLSP APIを通じて以下の構造情報を一括で取得する。

- ルート関数のcallers（BFS再帰探索による呼び出し元の網羅的取得）
- ルート関数のcallees（呼び出し先一覧）
- 仮想関数ディスパッチ（base method、dispatch callers、sibling overrides）
- 型階層（supertypes / subtypes）
- 各直接呼び出し元のhover情報（シグネチャ）とcallees

これらはすべて「誰が誰を呼んでいるか」という**構造的な接続関係**であり、clangdのインデックスから機械的に取得できる。ここまでは問題ない。

## 自動化できない部分：「影響は実際に伝播するのか」

問題は次のステップだ。`investigate`が「関数AはBを呼んでいる」と教えてくれても、修正の影響がB経由で実際に伝播するかは、別の判断が必要になる。

具体例を見てみよう。

```cpp
int caller() {
    int result = foo.calculate();  // ← investigateはここを見つける
    int b = result * 2;            // resultがbに伝播（前方スライス判定が必要）
    network.send(b);               // bがsendに渡される → 影響伝播あり
    logger.log("done");            // resultと無関係 → 影響なし
}
```

clangdは「`caller`が`calculate`を呼んでいる」ことは教えてくれるが、戻り値が`caller`内部でどう使われるかは教えてくれない。これはclangdの品質の問題ではなく、LSP仕様に関数内データフローを公開するプロトコルがそもそも定義されていないためだ。

この判定をさらに詳しく見ると、3つのステップに分かれる。

**ステップ1：データ依存の追跡。** callerのコードを読み、修正メソッドの戻り値がどの変数に代入され、どの文で使用され、どの関数に渡されるかを追跡する。

**ステップ2：制御依存の追跡。** 修正の影響を受ける値が条件分岐（if文やswitch文）の条件式に使われている場合、その分岐先の両方を影響範囲に含める必要がある。

**ステップ3：関数境界の漏洩チェック。** 各callerについて、影響が関数外に「漏洩」するかを判定する。漏洩経路は5つある——戻り値、メンバ変数への書き込み、参照/ポインタ引数への書き込み、グローバル/静的変数への書き込み、コールバック/observerの発火。漏洩が検出された場合、さらに上位のcallerに対してステップ1〜3を再帰的に繰り返す。

この一連の判定が「前方スライス」であり、現在はLLMがコードを読んで実行している部分だ。

```
investigate（自動化済）          LLM（判断）
─────────────────           ───────────────────────────
呼び出しグラフ取得     →    各callerのコードを読む
型階層取得             →    データ依存を追跡
シグネチャ取得         →    制御依存を追跡
仮想関数ディスパッチ   →    関数境界の漏洩を判定
                            追跡打ち切り/継続を判断
```

## なぜ完全な自動化が数学的に不可能なのか

### Riceの定理：1953年に証明された壁

前方スライスの完全な自動化が不可能であることは、数学的に証明されている。これはHenry Gordon Riceが1953年に証明した**Riceの定理**（Rice, 1953. "Classes of Recursively Enumerable Sets and Their Decision Problems." *Transactions of the American Mathematical Society*, Vol. 74, No. 2, pp. 358-366）から導かれる。

Riceの定理を平たく言えばこうだ。

> **プログラムの「振る舞いに関する性質」で、自明でないものは、すべて自動判定が不可能（決定不能）である。**

「自明でない」とは「すべてのプログラムに当てはまるわけでも、どのプログラムにも当てはまらないわけでもない」という意味だ。「このプログラムは停止するか」「この関数の出力は常に正か」「この変数の値は文Xに到達するか」——いずれも自明でない意味論的性質であり、一般に決定不能となる。

前方スライスで求めたい「ある文の出力値が、プログラム中のどの文に到達するか」は、まさにこの種の性質だ。Riceの定理は停止問題（Halting Problem）の決定不能性を一般化したものであり、前方スライスの決定不能性は停止問題に直接帰着できる。ある文`S`が到達可能かどうかを決定するには、`S`に至るパス上のすべての条件分岐とループが特定の入力で実行されるかを判定する必要があり、これは「プログラムが停止するか」と同等以上に困難だ。

これはツールの実装品質とは無関係な、計算理論の基本的帰結である。

### 具体的に何が決定できないか

C++で静的に（プログラムを実行せずに）決定できないケースを見てみよう。

```cpp
void example(Base* p, int userInput) {
    int a = foo.calculate();

    // (1) 実行時にしか分からない分岐
    if (userInput > 0) {
        send(a);     // 実行されるか？入力次第
    }

    // (2) 実行時にしか分からないディスパッチ
    p->process(a);   // どの実装が呼ばれるか？実行時の型次第

    // (3) 実行時にしか分からないポインタ
    int* ptr = array[userInput];
    *ptr = a;        // どのメモリに書いているか？入力次第
}
```

(1)では`userInput`の値がわからない限り`send(a)`が実行されるか決定できない。(2)では`p`の実行時型がわからない限りどのオーバーライドが呼ばれるか決定できない。(3)では`userInput`の値がわからない限りポインタの指す先が決定できない。

これらはC++特有の問題ではなく、チューリング完全なすべてのプログラミング言語に共通する本質的限界だ。

## 近似の2つの方向：「見落としゼロ」か「過剰検出ゼロ」か

完全な前方スライスが不可能であるなら、すべての実用的手法は**近似**にならざるを得ない。ここで決定的に重要なのは、近似の**方向**だ。

**Sound（健全）な近似**は、実際の影響範囲を必ず包含する方向に近似する。つまり偽陽性（過剰検出：実際には影響がないのに「影響あり」と報告する）はあるが、偽陰性（見落とし：実際に影響があるのに「影響なし」と報告する）はゼロを保証する。

**Complete（完全）な近似**は逆方向で、報告されたものは必ず実際に影響があることを保証する。偽陰性（見落とし）はあり得るが、偽陽性（過剰検出）はゼロ。

Riceの定理の帰結として、SoundとCompleteを同時に達成することは不可能だ。影響範囲調査においては見落としが最も危険（リリース後にバグとして顕在化する）であるため、sound（見落としゼロ）の方向が望ましい。

```
実際の影響範囲:              {A, B, C}

Soundなツールの出力:         {A, B, C, D, E, F}
                             ← 過剰だが見落としゼロを保証

LLMの出力:                   {A, B, D}
                             ← Cを見落とし、Dを誤検出
                             ← どちらの方向にも間違える
                             ← いつ間違えるか予測不能
```

## 「ほぼ健全（Soundy）」という現実——業界は何を妥協しているのか

ここで、理論と実務の間に横たわる重要なギャップに触れなければならない。

2015年にLivshits, Sridharan, Smaragdakisら10名の研究者が発表した論文 "In Defense of Soundiness: A Manifesto"（*Communications of the ACM*, Vol. 58, No. 2, pp. 44-46）は、静的解析コミュニティに衝撃を与えた。彼らの主張の核心はこうだ。

> **我々が知る限り、バグ検出・リファクタリング支援・プログラミング自動化等に広く使われている現実的な全プログラム解析ツールで、意図的にunsound（不健全）な選択を行わないものは一つも存在しない。**

彼らはこの状態を**「soundy（ほぼ健全）」** と名付けた。soundyな解析とは、標準的な制御フロー・関数呼び出し・ヒープのLOAD/STOREなど「コンパイラの中間表現で見える基本的な操作」はsound（健全）に扱い、リフレクション・動的ロード・インラインアセンブリなど「特定の高度に動的な言語機能」のみをunsound（不健全）に扱う解析のことだ。

つまり、soundnessの妥協は例外ではなく**業界標準**だ。Coverity、Fortify、GrammaTech、IBMなどの商用ツールも、学術論文で提案される解析手法も、実用に耐える精度とスケーラビリティを達成するために、意図的にsoundnessを放棄している。

この事実は、C++に限った話ではない。Javaの解析でもリフレクションや動的クラスローディングはunsoundに扱われるのが一般的であり、JavaScriptの`eval`はほぼすべてのツールで無視される。問題は、**C++ではunsoundに扱わざるを得ない言語機能の範囲が他の言語よりも格段に広い**ことだ。

## C++ではなぜ特に困難なのか

### コンパイラは答えを「知っている」——だが教えてくれない

まず一つの皮肉な事実がある。clang（clangdの基盤となるコンパイラ）はコンパイル時に、AST → CFG → データフロー解析 → 最適化 → コード生成という過程でデータフローを完全に把握している。前方スライスに必要な情報はコンパイラの内部に存在する。

問題は、この情報をclangdのLSP（Language Server Protocol）APIが外部に公開していないことだ。LSP仕様にはhover、references、callHierarchyなどの構造的クエリは含まれるが、「ある文を起点とした前方スライスを返す」というプロトコルは定義されていない。これは単なる仕様の不備ではなく、後述するスケーラビリティの問題から来る意図的な設計判断だ。

### Sound（見落としゼロ）を目指すとC++では何が起きるか

Sound（偽陰性ゼロ）な前方スライスは、C++に対しても原理的には構築可能だ。しかしC++の言語特性が**過大近似を爆発的に増大**させ、解析結果を実質的に無意味にする。

Soundな解析は「分からないものは安全側に倒す」方針を取る。たとえば「このポインタがどこを指すか分からない」場合、「プログラム中のどのメモリでも指しうる」と仮定する。C++では以下の要因により「分からない」ケースが頻出し、この過大近似が雪だるま式に膨らむ。

**生ポインタとポインタ演算。** `*(p + offset)`が指す先を静的に限定できない。Balatsouras & Smaragdakis (2016) はこの問題を端的に指摘している。「Cのポインタエイリアス解析は厳密にsoundであることが不可能である。さもなければ、メモリ上のほとんどの位置が任意のメモリ位置を指し得ると結論づけることになるからだ」（"Structure-Sensitive Points-To Analysis for C and C++," *SAS 2016*）。

**`reinterpret_cast` / `void*`。** 型安全性の保証がない型変換は、型ベースのエイリアス解析（TBAA: Type-Based Alias Analysis）を破壊する。

**`union`とプレースメント`new`。** 同じメモリ領域を異なる型で解釈できる`union`や、任意のメモリ位置にオブジェクトを構築できるプレースメント`new`は、ポインタ解析の基本的な前提——「オブジェクトは個別のメモリ空間に割り当てられ、生涯を通じて単一の型を持つ」——を崩す。

**テンプレート実体化。** テンプレートはコンパイル時にコード量を爆発的に増大させる。解析対象のコードが文字通り何倍にも膨らむ。

**多重継承＋仮想関数。** 仮想ディスパッチ先の候補が多くなり、ポインタ解析の精度が低い場合、あらゆるオーバーライドが候補として残る。

**プリプロセッサマクロ。** テキスト置換による非構造的なコード変形であり、マクロ展開後のコードに対する解析は、ソースコード上の推論と大きく乖離する。

**インラインアセンブリ。** 解析不能。

**例外処理。** `try-catch`による非局所的な制御フローの跳躍は、前方スライスが追跡すべきパスの数を爆発させる。

結果として、C++でsound（見落としゼロ）な前方スライスを実行すると、プログラムのほぼ全体が影響範囲として報告される。「全部影響あります」では調査として役に立たない。

### 精度を上げるには計算量が爆発する

前方スライスの精度は、その基盤となる**ポインタ解析（Points-to Analysis）** の精度に完全に依存する。「変数`x`の値がどこに流れるか」を追跡するには、「ポインタ`p`がどのメモリを指しているか」を知る必要があるからだ。Wikipediaの "Pointer analysis" の項目にもあるとおり、完全に正確なポインタ解析は決定不能であり、実用的な手法はすべて近似である。

ポインタ解析には精度とスケーラビリティのトレードオフがある。

**Steensgaardの解析**（単一化ベース）はほぼ線形時間O(n)で動作し、巨大なコードベースにもスケールする。しかし精度が粗く、ポインタが一度でも同じ場所に代入されると、それらが指し得る対象を全部マージしてしまう。前方スライスの入力としては「ほぼ全体が影響範囲」という結果になりやすい。

**Andersenの解析**（包含制約ベース）は精度が高いがO(n³)の計算量を持ち、大規模コードベースではスケールしない。

**フロー感度あり＋コンテキスト感度あり**の解析はさらに精密だが、計算量は指数関数的に爆発する。

つまり、精度を上げようとすれば計算が終わらなくなり、スケーラビリティを優先すれば精度が崩壊する。このジレンマはC++に限らずすべての言語に存在するが、C++の言語特性がこのジレンマを極端に悪化させる。

## Chromiumという現実——3,600万行のC++

ここまでの議論を、現実の巨大C++コードベースに当てはめてみよう。

Chromiumブラウザのコードベースは、コメントと空行を除いて**約3,600万行のソースコード**を持つ（Wikipedia, "Chromium (web browser)" より）。C++がコードベースの半分以上を占め、BlinkレンダリングエンジンとV8 JavaScriptエンジン、HTTPプロトコル実装、内部キャッシュシステム、拡張API、UIの大部分がC++で書かれている。

### Googleですら全プログラム解析を日常的に実行していない

Chromiumの規模感を理解するために、Googleの静的解析に関する実態を見てみよう。

Sadowski et al. (2018) の "Lessons from Building Static Analysis Tools at Google"（*Communications of the ACM*, Vol. 61, No. 4, pp. 58-66）は、Google内部の静的解析の現状を率直に報告している。その記述は衝撃的だ。

> **Googleは、Google規模で手続き間（interprocedural）解析や全プログラム解析を実行するインフラ支援を持っていないし、高度な静的解析技術（例えばseparation logic）を大規模には使用していない。**

Googleのすべてのコードは単一のモノリシックなリポジトリに存在し、概念上はリポジトリ内の任意のコードが任意のバイナリの一部となりうる。したがって、あるコードレビューの解析結果を求めるだけで、リポジトリ全体の解析が必要になるシナリオが想定される。FacebookのInferが合成的解析（compositional analysis）でスケーラビリティを追求しているが、それでもGoogleの数十億行規模のリポジトリにスケールさせるには相当のエンジニアリング努力が必要だと論文は述べている。

さらに、PhASARフレームワークの博士論文 "Scaling Static Whole-Program Analysis to Modern C and C++ Software Development"（Schubert, 2023, University of Paderborn）は、この状況をより広い視点で確認している。

> **Googleはコードベースへの全コミットに対して高コストな手続き間（全プログラム）解析を実行することを断念している。代わりに、コード変更をまとめて1日の終わりにバッチスタイルで全プログラム解析を実行している。**

同論文はさらに踏み込んで、「C/C++を対象とする深い意味論的静的解析のスケーラブルなフレームワークは、実質的に存在しない」と結論づけている。

前方スライスは手続き間データフロー解析の一形態であり、まさにこの「Googleですら実行できていない」カテゴリに該当する。

### Chromiumプロジェクト自身の経験

Chromiumプロジェクトの開発者メーリングリスト（chromium-dev）での議論も、この現実を裏付けている。

静的解析に関する議論で、あるChromium開発者は次のように報告している。「以前はCoverityを使っていたが、偽陽性が多すぎた。Bruceの/analyze botは時々バグを見つけるが、これも偽陽性が多い。Clang Static Analyzerは一時期動いていたが、これも偽陽性が多い」。

別の開発者は、clang-tidyをChromiumのコードベースに適用するための作業を進めていることを報告しつつ、コンパイルデータベースの問題が主要な障壁であると述べている。

また、Bruce Dawsonの2020年の分析（"Big Project Build Times—Chromium," randomascii.wordpress.com）によれば、Chromiumの30,137個のソースファイルが合計1,160万行のソースコードを含む一方、ヘッダファイルのインクルードにより**処理される行数は累計36億行**に達する。メインのソースファイルは処理される全行数のわずか0.32%にすぎない。マクロやテンプレートを考慮した解析がいかに膨大になるかが分かる。

### スケーラビリティの壁が意味するもの

これらの事実は、clangdがデータフロー解析をLSP APIとして公開していない理由を明確に示している。IDE向けの言語サーバーは、開発者のキー入力の合間（ミリ秒〜秒単位）でレスポンスを返す必要がある。全プログラム的なデータフロー解析は、Chromium規模では分単位から時間単位の計算を要する。そのような解析をインタラクティブなAPIとして公開することは、設計上合理的でない。

## 既存ツールはこの問題をどう扱っているか

それでは、clangd以外のツールなら前方スライスを自動化できるのだろうか。主要な静的解析ツールを評価した結果を以下にまとめる。

### Coverity

Coverityはバグ検出を目的とした商用静的解析ツールで、手続き間データフロー解析を中核技術とする。ポインタエイリアス・仮想関数ディスパッチ・テンプレート実体化を含めた精密な追跡を行い、null dereference、resource leak、buffer overflow等を検出する。

しかしCoverityの設計目標は「既知のバグパターンの検出」であり、**「任意の関数を起点とした前方スライスの結果を出力する」というインターフェースは提供していない**。検出したバグのレポートに付随する「汚染経路（taint path）」としてデータフロー追跡の結果が断片的に見えるが、汎用の前方スライスクエリとしては使えない。

前述のとおり、Chromiumプロジェクトでは偽陽性の多さから運用が中断された経緯がある。

### CodeQL

GitHub/Semmle社のCodeQLは、ソースコードからリレーショナルデータベースを構築し、Datalog風のクエリ言語でデータフロー・制御フローを問い合わせるツールだ。taint trackingクエリを記述することで、前方スライスに近い解析を実行できる。CodeQLの公式ドキュメントによれば、ローカルデータフロー（関数内）とグローバルデータフロー（関数間）の両方をサポートし、taint trackingではデータ値が厳密に保存されないステップ（例：`y = x + 1`で`x`のtaintが`y`に伝播する）も追跡できる。

現存するツールの中で最も柔軟なアプローチだが、実用上の制約は大きい。

**source/sinkの事前指定が必須。** CodeQLの公式トレーニング資料が明確に述べているとおり、「ローカルデータフローはデータベース内の全関数に対して計算可能だが、グローバルデータフローはパス数が指数関数的に大きくなるため不可能」である。そのため、クエリ作成者がソース（汚染源）とシンク（検出対象）を指定し、探索空間を限定する必要がある。「任意の起点からすべての到達可能文を列挙する」という汎用前方スライスを、sourceを`any()`に設定して実行しようとする試みは、GitHubのCodeQL Discussions #6239で実際に試みられ、期待通りに動作しなかった。

**C++のエイリアス処理に既知の制限がある。** GitHub CodeQL Discussions #9863では、C++のポインタエイリアシング存在下でtaintが消失するケースが報告されている。CodeQLの開発者自身が「IRデータフローライブラリは相反する優先事項を持っている可能性があり、容易に変更できるとは限らない」と回答している。

**構造体ポインタを介したtaint伝播に制限がある。** GitHub CodeQL Issues #7053では、構造体ポインタを関数間で渡した際にフィールドのtaintが追跡できないケースが報告されている。

加えて、影響範囲調査用のカスタムクエリの設計・実装・チューニングは非自明な作業であり、プロジェクト固有の知識が必要になる。

### Clang Static Analyzer

clangのASTとCFG上でパス感度あり（path-sensitive）のシンボリック実行を行う。カスタムチェッカーをC++ APIを使って書けばデータフロー情報に直接アクセスできるが、チェッカー開発は非自明な作業である。

最大の実用上の制約は**パス爆発（path explosion）** だ。プログラム中の分岐点が増えるほど探索すべきパスの数が指数関数的に増大し、解析が打ち切られる。Chromiumの公式ドキュメント（"The Clang Static Analyzer," chromium.googlesource.com）でも、Clang Static Analyzerは利用可能であるが、有効なチェッカーはcore、cplusplus、deadcodeの3スイート（それ以外は偽陽性やプラットフォーム不一致のため除外）に限定されていると記されている。

### Joern

ソースコードをCPG（Code Property Graph：AST＋CFG＋PDGの統合グラフ）に変換し、グラフクエリでスライスを実行できる。CPG上でデータ依存・制御依存をたどるクエリを書けば前方スライスに相当する解析が可能だ。ただし、C++対応の成熟度はJava/JavaScript対応より低い。

### CppDepend / Understand

関数レベルの依存グラフを出力可能だが、文レベルのデータフローは限定的。前方スライスに必要な粒度の解析は提供していない。

### ツール比較のまとめ

| ツール | 内部データフロー解析 | 汎用前方スライスAPI | C++対応 |
|--------|:---:|:---:|:---:|
| Coverity | あり（手続き間） | なし（バグ検出用のみ） | あり |
| CodeQL | あり（taint tracking） | 部分的（source/sink指定が必須） | あり（制限あり） |
| Clang Static Analyzer | あり（パス感度） | なし（チェッカー開発が必要） | あり |
| Joern | あり（CPGクエリ） | あり（クエリ開発が必要） | あり（成熟度低） |
| CppDepend / Understand | 部分的（関数レベル） | なし | あり |
| clangd | コンパイラ内部では実行 | なし（LSP未公開） | あり |

## C++以外の言語なら解決するのか？——規模が支配する壁

「C++が難しいのは分かった。ではJavaやGoやRustなら、Chromium規模でも前方スライスを自動化できるのか？」

答えは**「言語特性により困難度は異なるが、規模そのものが支配的な制約となるため、容易ではない」** だ。

### Java：解析しやすいが、規模とリフレクションの壁がある

Javaはポインタ演算がなく、ガベージコレクションがあり、単一継承である。これによりポインタ解析の精度はC++より大幅に向上する。FlowDroid、Soot、TAJ等のフレームワークがAndroidアプリ（数万〜数十万行）で成功を収めている。

しかし、Chromium規模（3,600万行）のJavaプログラムを仮定した場合、以下の問題が顕在化する。

**リフレクション。** `Class.forName()`、`Method.invoke()`等のリフレクションAPIは、コールグラフを静的に構築できなくする。Livshitsらの「soundiness」論文は、リフレクションをJavaにおけるunsoundnessの最大の原因として挙げている。大規模Javaアプリケーション（例えばEclipse）ではリフレクションが遍在しており、これを無視すると解析がプログラムの大部分に到達できなくなる。

**動的クラスローディング。** 実行時にロードされるクラスは静的解析時に存在すら不明であり、これらを経由するデータフローは追跡不能。

**フレームワーク依存。** Spring、Struts、Hibernate等のDIフレームワークはリフレクションとXML設定ファイルで制御フローを構成する。フレームワーク固有のモデリングなしでは、アプリケーションの入口点すら特定できない。TAJ（IBM Research, 2009）やF4F（Sridharan et al., 2011）はフレームワーク対応のtaint解析を提案しているが、フレームワークごとにカスタム対応が必要となる。

**スケーラビリティ。** ICSE 2023で発表された論文 "Scalable Compositional Static Taint Analysis for Large-Scale Industrial Microservices"（Sui et al.）は、大規模マイクロサービス（数十万行規模）に対する既存のtaint解析ツールのスケーラビリティが不十分であることを報告している。コンテキスト感度ありの全プログラムポインタ解析は、DaCapoベンチマーク（数万〜数十万行程度）ですら数分から数十分を要する。

### Go：比較的解析しやすいが、並行性の壁がある

Goはインターフェースの動的ディスパッチが限定的で、リフレクションの使用頻度もJavaより低い傾向がある。しかし以下の問題がある。

**ゴルーチンとチャネル。** 並行プログラムのデータフロー追跡では、チャネルを介したメッセージパッシングのモデル化が必要であり、これは全プログラム解析を要する。

**空インターフェース。** `interface{}`（Go 1.18以降は`any`）は任意の型を受け入れ、型情報を消去するため、C++の`void*`と類似の問題を引き起こす。

**`unsafe`パッケージ。** `unsafe.Pointer`はC++のポインタ演算と同等の問題を生む。

### Rust：所有権が助けるが、万能ではない

Rustの所有権システムは`&mut`の排他性を言語レベルで保証し、エイリアスの制約を大幅に強化する。これはポインタ解析を簡素化し、前方スライスの精度向上に直結する。

しかし以下の制約がある。

**`unsafe`ブロック。** Chromium規模のコードベースではFFI（Foreign Function Interface）や低レベル操作で`unsafe`が必要になり、そこでは所有権の保証が失われる。

**トレイトオブジェクト（`dyn Trait`）。** 仮想ディスパッチに相当し、動的ディスパッチ先の候補を静的に絞る必要がある。

**非同期ランタイム。** `async`/`await`のコンパイル時の状態機械変換後のコードに対する解析は複雑であり、ソースコード上の推論と乖離する。

### 言語×規模のマトリクス

| 言語 | 小規模（〜10万行） | 中規模（〜100万行） | 大規模（1,000万行超） |
|------|:---:|:---:|:---:|
| C++ | 困難 | 非常に困難 | 実質不可能（sound） |
| Java | 実用的 | 困難 | 非常に困難 |
| Go | 比較的実用的 | 困難 | 困難 |
| Rust | 比較的実用的 | 実用的（unsafe少なら） | 未知 |

重要な結論は、**1,000万行を超えると言語の特性差よりもスケーラビリティの壁が支配的になる**ことだ。Javaのような「解析しやすい」言語であっても、Googleは大規模な手続き間解析を日常的には実行していない。言語を変えてもChromium規模の前方スライスが「簡単」になることはない。

## 近似の性質を比較する

ここまでの議論を踏まえ、各手法の近似の性質を比較してみよう。

| 解析手法 | 偽陽性（過剰検出） | 偽陰性（見落とし） | 再現性 |
|---------|:---:|:---:|:---:|
| Sound静的解析（理論上） | あり | なし（保証） | あり |
| 実用的なC++静的解析ツール | あり（抑制済） | あり（soundness妥協） | あり |
| LLM | あり | あり（予測不能） | なし |
| 人間によるコードレビュー | あり | あり | なし |

C++においては、実用的な静的解析ツールもsoundnessを保証できない。したがって、ツールとLLMの差は「偽陰性の有無」ではなく**「偽陰性の予測可能性と再現性」** に縮小する。ツールの偽陰性は既知のパターン（特定の言語機能を扱えないなど）に限定され再現性があるが、LLMの偽陰性は予測不能であり同じコードに対して異なる判定を返しうる。

## clangd-cliプロジェクトにおける現実的選択

以上の分析を踏まえ、clangd-cliプロジェクトでは以下の分業を採用している。

- **データ収集**：clangd-cliの`investigate`コマンドで自動化
- **前方スライス判定**：LLMが担当

この選択の根拠は3つある。

**即時利用可能性。** LLMは追加のツール導入やクエリ開発なしで、前方スライス判定を今すぐ実行できる。CodeQLのクエリ開発やClang Static Analyzerのカスタムチェッカー開発は非自明な作業であり、プロジェクト固有のチューニングも必要になる。

**実用的な精度。** C++においてはツールもLLMもsoundnessを保証できない以上、LLMによる判定は実務上合理的な選択肢である。ただし、LLMの判定精度について定量的なベンチマークは今後の課題として残る。

**開発コストの回避。** 前方スライスの自動化ツールを開発・維持するコストは大きい。CodeQLクエリの開発には中〜高程度の工数がかかり、Clang Static Analyzerのカスタムチェッカーに至ってはC++ APIを用いた高度な開発が必要になる。

### LLMによる判定の限界

この選択には明確な限界がある。

**Soundnessの保証なし。** 見落としが発生しうるが、いつ発生するか予測できない。

**再現性なし。** 同じコードに対して異なる判定を返しうる。

**コンテキスト窓の制約。** 大規模な関数や深い呼び出しチェーンでは、すべてのコードを一度に読めない場合がある。

これらの限界を前提として、影響範囲調査ガイドでは「不確実性が残る変更では、安全側に再テスト範囲を拡大する」という運用方針を定めている。

## 将来的な改善の方向性

前方スライス判定の精度向上・自動化には以下の方向性がありうる。

**CodeQLクエリの開発。** taint trackingクエリで前方スライスに近い解析を実行する。source/sinkの設計が鍵であり、影響範囲調査に特化したクエリパターンの確立が必要。開発コストは中〜高。

**Clang Static Analyzerカスタムチェッカー。** clangのCFG/データフロー情報に直接アクセスし、前方スライス専用のチェッカーを開発する。最も精密な解析が可能だが、パス爆発の制御を含め開発コストは高い。

**clangdへのforward-slice API追加。** upstream貢献としてLSP仕様の拡張を提案する。技術的には実現可能だが、LSPコミュニティでの合意形成と受理の保証がなく、開発コストは高い。

**LLM＋静的解析のハイブリッド。** 静的解析ツールでsound（過大近似含む）な候補集合を生成し、LLMがその中から実際に影響のあるものを絞り込む。過大近似の削減をLLMに委ね、見落としリスクを静的解析側で担保する構成。最も現実的な改善方向であり、開発コストは中程度。

## まとめ

コード変更の影響範囲を完全に自動追跡することは、1953年にRiceが証明した数学的事実により、原理的に不可能だ。すべての実用的手法は近似であり、その近似の方向（見落としゼロを優先するか、過剰検出ゼロを優先するか）が解析の性質を決定する。

C++ではポインタ演算、テンプレート、マクロ、多重継承、例外処理等の言語特性が近似の爆発を引き起こし、sound（見落としゼロ）な解析は「プログラム全体が影響範囲」という実質無意味な結果を返す。実用的なツールはsoundnessを意図的に妥協しており、Livshitsらが2015年に指摘したように、これは例外ではなく業界標準だ。

Chromium規模（3,600万行）のコードベースでは、スケーラビリティの壁が言語特性の壁に加わる。Googleですら手続き間・全プログラム解析を日常的には実行していない。この制約はC++に限らず、JavaやGoであってもChromium規模では深刻な問題となる。

clangd-cliプロジェクトでは、機械的に取得可能な構造情報（呼び出しグラフ、型階層、仮想関数ディスパッチ）の収集を自動化し、前方スライス判定をLLMに委ねる分業構造を採用している。これは理論的・実用的制約を踏まえた現実的な選択であり、将来的にはLLMと静的解析のハイブリッドアプローチによる改善が期待される。
