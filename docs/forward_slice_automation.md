# 前方スライス自動化の技術的限界と現実的選択

## 1. 本資料の位置づけ

[影響範囲調査ガイド](./impact_analysis_guide.md)では、前方スライス（Forward Slice）を影響分析の基本骨格として用いている。本資料は、前方スライスの自動化に関する技術的限界と、clangd-cliプロジェクトにおける現実的な選択について整理する。

---

## 2. 影響範囲調査における分業構造

影響範囲調査は「データ収集」と「前方スライス判定」の2段階に分かれる。

### 2.1 データ収集（自動化済）

clangd-cliの`investigate`コマンドは、clangdのLSP APIを通じて以下の構造情報を一括取得する：

- ルート関数のcallers（BFS再帰探索）
- ルート関数のcallees
- 仮想関数ディスパッチ（base method、dispatch callers、sibling overrides）
- 型階層（supertypes / subtypes）
- 各直接呼び出し元のhover（シグネチャ）とcallees

これらは「誰が誰を呼んでいるか」という**構造的な接続関係**であり、clangdのインデックスから機械的に取得できる。

### 2.2 前方スライス判定（LLMが担当）

`investigate`が返した各callerについて、**影響が実際に伝播するかどうか**を判定する必要がある。具体的には影響範囲調査ガイドのSTEP 3〜5に対応する：

**STEP 3（データ依存の追跡）**：callerのコードを読み、修正メソッドの戻り値がどの変数に代入され、どの文で使用され、どの関数に渡されるかを追跡する。

```cpp
int caller() {
    int result = foo.calculate();  // investigate: "callerがcalculateを呼んでいる"
    int b = result * 2;            // 前方スライス判定: "resultがbに伝播"
    network.send(b);               // 前方スライス判定: "bがsendに渡される → 影響伝播"
    logger.log("done");            // 前方スライス判定: "resultと無関係 → 影響なし"
}
```

clangdは「callerがcalculateを呼んでいる」ことは教えてくれるが、戻り値がcaller内部でどう使われるかは教えてくれない。これはclangdのLSP APIに関数内データフローを公開するインターフェースが存在しないためである。

**STEP 4（制御依存の追跡）**：修正の影響を受ける値が条件分岐に使われている場合、両方の分岐先を影響範囲に含める。

**STEP 5（関数境界の漏洩チェック）**：各callerについて、影響が5つの経路（戻り値、メンバ変数、参照/ポインタ引数、グローバル/静的変数、コールバック/observer）を通じて関数外に漏洩するかを判定し、漏洩がある場合はさらに上位のcallerに対して再帰的にSTEP 3〜5を繰り返す。

### 2.3 分業の構造

```
investigate（自動）          LLM（手動判断）
─────────────────           ───────────────────────────
呼び出しグラフ取得     →    各callerのコードを読む
型階層取得             →    データ依存を追跡
シグネチャ取得         →    制御依存を追跡
仮想関数ディスパッチ   →    関数境界の漏洩を判定
                            追跡打ち切り/継続を判断
```

## 3. なぜ前方スライスの自動化が困難か

### 3.1 コンパイラは内部的にデータフローを把握している

clang（clangdの基盤）はコンパイル時に AST → CFG → データフロー解析 → 最適化 → コード生成 の過程でデータフローを完全に把握している。前方スライスに必要な情報はコンパイラ内部に存在する。

問題は、この情報をclangdのLSP APIが**外部に公開していない**ことにある。LSP仕様にはhover、references、callHierarchyなどの構造的クエリは含まれるが、「ある文を起点とした前方スライスを返す」というプロトコルは定義されていない。

### 3.2 理論的限界：完全な前方スライスは不可能

仮にLSP APIの制約がなくても、完全な前方スライスは理論的に不可能である。これはRiceの定理（1953）から導かれる数学的事実であり、ツールの実装品質とは無関係である。

**Riceの定理の帰結**：プログラムの非自明な意味的性質は、一般に決定不能（undecidable）である。あるプログラム点から到達可能なすべての文の集合を正確に求めることは、停止問題と同等の困難さを持つ。

具体的にC++で静的に決定できないケースは以下の通りである：

```cpp
void example(Base* p, int userInput) {
    int a = foo.calculate();

    // (1) 実行時にしか分からない分岐
    if (userInput > 0) {
        send(a);     // 実行されるか？入力次第
    }

    // (2) 実行時にしか分からないディスパッチ
    p->process(a);   // どの実装が呼ばれるか？実行時の型次第

    // (3) 実行時にしか分からないポインタ
    int* ptr = array[userInput];
    *ptr = a;        // どのメモリに書いているか？入力次第
}
```

### 3.3 静的解析は近似であり、近似の方向が重要

あらゆる静的解析は近似にならざるを得ないが、近似の**方向**によって性質が大きく異なる。

| 近似方向 | 定義 | 結果 |
|---------|------|------|
| **Sound（健全）** | 実際の影響範囲を必ず包含する | 偽陽性（過剰検出）はあるが、偽陰性（見落とし）はゼロ |
| **Complete（完全）** | 報告されたものは必ず実際に影響がある | 偽陰性（見落とし）はあるが、偽陽性（過剰検出）はゼロ |

SoundとCompleteの両立は不可能である（Riceの定理の帰結）。影響範囲調査においては見落としが最も危険であるため、sound（健全）な近似が望ましい。

```
実際の影響範囲:              {A, B, C}

Soundなツールの出力:         {A, B, C, D, E, F}
                             ← 過剰だが見落としゼロを保証

LLMの出力:                   {A, B, D}
                             ← Cを見落とし、Dを誤検出
                             ← どちらの方向にも間違える
                             ← いつ間違えるか予測不能
```

## 4. C++における特有の困難

### 4.1 Soundな解析は可能だが、C++では実用性が低い

Sound（偽陰性ゼロ）な前方スライスは、C++に対しても原理的には構築可能である。しかしC++の言語特性が過大近似を爆発的に増大させ、解析結果の実用性を著しく低下させる。

Soundな解析は「分からないものは安全側に倒す」方針を取る。たとえば「このポインタがどこを指すか分からない」場合、「プログラム中のどのメモリでも指しうる」と仮定する。C++では以下の要因により「分からない」ケースが頻出し、過大近似が爆発する：

- **生ポインタとポインタ演算**：`*(p + offset)` が指す先を静的に限定できない
- **reinterpret_cast / void***：型安全性の保証がない型変換
- **テンプレート実体化**：コード量が爆発的に増大
- **多重継承 + 仮想関数**：ディスパッチ先の候補が多い
- **プリプロセッサマクロ**：テキスト置換による非構造的なコード変形
- **インラインアセンブリ**：解析不能

| 言語 | soundに解析した場合の過大近似 |
|------|--------------------------|
| Java | 管理可能な量（ポインタ演算なし、GC、単一継承） |
| Go | 管理可能な量（ポインタ演算なし、インターフェース限定的） |
| C++ | プログラムのほぼ全体が影響範囲になりうる |

### 4.2 実用的なC++解析ツールはsoundnessを妥協している

「前方スライスの結果がプログラム全体」では調査として役に立たない。そのため、C++向けの実用的な静的解析ツールはsoundnessを**意図的に妥協**し、偽陰性（見落とし）の可能性を受け入れる代わりに偽陽性（過剰検出）を減らしている。

結果として、C++においては静的解析ツールもLLMも、**どちらも偽陰性がありうる**。近似の性質は異なるが（ツールは再現性があり近似方向が既知、LLMは再現性がなく近似方向が未知）、偽陰性ゼロの保証はどちらも持たない。

## 5. 既存ツールの評価

### 5.1 前方スライスの観点での分類

| ツール | 内部でデータフロー解析を行っているか | 「任意の起点から前方スライスを出せ」APIがあるか | C++対応 |
|--------|--------------------------------------|-----------------------------------------------|---------|
| **Coverity** | はい（手続き間データフロー解析が中核技術） | なし — バグ検出用のAPIのみ | はい |
| **CodeQL** | はい（taint trackingをクエリとして記述可能） | あり — ただしクエリ開発が必要 | はい |
| **Clang Static Analyzer** | はい（パス感度ありのシンボリック実行） | なし — カスタムチェッカー開発が必要 | はい |
| **Joern** | はい（CPG: Code Property Graph上でクエリ実行） | あり — ただしクエリ開発が必要 | はい（C/C++対応） |
| **CppDepend / Understand** | 部分的（関数レベルの依存グラフ） | 依存グラフは出力可能 — 文レベルのデータフローは限定的 | はい |
| **clangd** | コンパイラ内部では行っている | なし — LSP APIで公開されていない | はい |

### 5.2 各ツールの詳細

**Coverity**：手続き間データフロー解析を中核技術とし、ポインタエイリアス・仮想関数ディスパッチ・テンプレート実体化を含めた精密な追跡を行う。ただし目的はバグ検出（null dereference、resource leak、buffer overflow等）であり、「任意の関数を起点とした前方スライスの結果を出力する」というインターフェースは提供していない。検出したバグのレポートに付随する「汚染経路（taint path）」として、データフロー追跡の結果が断片的に見えるのみである。

**CodeQL**：ソースコードからリレーショナルデータベースを構築し、Datalog風のクエリ言語でデータフロー・制御フローを問い合わせる。taint trackingクエリを記述することで、前方スライスに近い解析を実行できる。最も柔軟なアプローチだが、影響範囲調査用のクエリを設計・実装・チューニングする開発コストがかかる。

**Clang Static Analyzer**：clangのASTとCFGの上でパス感度あり（path-sensitive）のシンボリック実行を行う。カスタムチェッカーを書けばデータフロー情報に直接アクセスできるが、チェッカーの開発はC++ APIを用いた非自明な作業である。パス爆発（path explosion）による解析打ち切りが実用上の制約となる。

**Joern**：ソースコードをCPG（Code Property Graph：AST + CFG + PDGの統合グラフ）に変換し、グラフクエリでスライスを実行できる。CPG上でデータ依存・制御依存をたどるクエリを書けば前方スライスに相当する解析が可能。ただしC++対応の成熟度はJava/JavaScript対応より低い。

### 5.3 近似の性質の比較

| 解析手法 | 偽陽性（過剰検出） | 偽陰性（見落とし） | 再現性 |
|---------|-------------------|-------------------|--------|
| Sound静的解析（理論上） | あり | **なし（保証）** | あり |
| 実用的なC++静的解析ツール | あり（抑制済） | **あり（soundnessを妥協）** | あり |
| LLM | あり | **あり（予測不能）** | なし |
| 人間によるコードレビュー | あり | あり | なし |

C++においては、実用的な静的解析ツールもsoundnessを保証できないため、ツールとLLMの差は「偽陰性の有無」ではなく「偽陰性の予測可能性と再現性」に縮小する。

## 6. clangd-cliプロジェクトにおける選択

### 6.1 現状の選択とその根拠

clangd-cliプロジェクトでは、以下の分業を採用している：

- **データ収集**：clangd-cliの`investigate`コマンドで自動化
- **前方スライス判定**：LLMが担当

この選択の根拠は以下の通りである：

1. **即時利用可能性**：LLMは追加のツール導入やクエリ開発なしで前方スライス判定を実行できる
2. **実用的な精度**：LLMの判定精度は人間のコードレビューと同程度であり、影響範囲調査の実務に十分な水準である
3. **開発コストの回避**：CodeQL/Joernのクエリ開発やClang Static Analyzerのカスタムチェッカー開発は非自明な作業であり、プロジェクト固有のチューニングも必要になる

### 6.2 LLMによる前方スライス判定の限界

LLMによる前方スライス判定には以下の限界がある：

- **soundnessの保証なし**：見落としが発生しうるが、いつ発生するか予測できない
- **再現性なし**：同じコードに対して異なる判定を返しうる
- **コンテキスト窓の制約**：大規模な関数や深い呼び出しチェーンでは、すべてのコードを一度に読めない場合がある

影響範囲調査ガイド（6.4節）が「不確実性が残る変更では、安全側に再テスト範囲を拡大する」と記述しているのは、この限界を前提とした運用方針である。

### 6.3 将来的な改善の方向性

前方スライス判定の精度向上・自動化には以下の方向性がありうる：

| アプローチ | 概要 | 開発コスト |
|-----------|------|-----------|
| CodeQLクエリの開発 | taint trackingクエリで前方スライスに近い解析を実行 | 中〜高 |
| Clang Static Analyzerカスタムチェッカー | clangのCFG/データフロー情報に直接アクセス | 高 |
| clangdへのforward-slice API追加 | upstream貢献として提案 | 高（受理の保証なし） |
| LLM + 静的解析のハイブリッド | ツールでsoundな候補を出し、LLMで絞り込む | 中 |
